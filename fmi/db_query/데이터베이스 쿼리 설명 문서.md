# FMI MySQL 데이터베이스 구축 및 데이터 적재 가이드

이 문서는 팀원이 동일한 방식으로 MySQL 데이터베이스를 생성하고, 임상도 CSV 데이터를 적재할 수 있도록 작성한 실행 가이드입니다. 

## 1. 프로젝트 폴더 구조

현재 프로젝트는 아래 구조를 기준으로 합니다.

```text
fmi/
├─ db_query/
│  ├─ 데이터베이스 쿼리 설명 문서.md
│  ├─ create_database.py
│  └─ load_data.py
│
├─ forest_fmi.csv
├─ 임상도.md
└─ .env
```

## 2. 각 파일 역할

### `fmi/db_query/create_database.py`

MySQL 데이터베이스와 테이블을 생성하는 코드입니다.

생성되는 테이블은 총 1개입니다.

```text
forest_fmi   임상도 및 임상지수(FMI) — 강원도 562,276 구역
```

기본키는 다음과 같습니다.

```text
forest_fmi:
polygon_id
```

### `db_query/load_data.py`

CSV 파일을 읽어서 MySQL 테이블에 데이터를 적재하는 코드입니다.

적재 대상 파일은 다음과 같습니다.

```text
forest_fmi.csv
```

전체 562,276행을 필터링 없이 그대로 적재합니다.

`FMI == 0`인 행(무립목지/비산림)도 DB에는 적재되며, 분석 시 쿼리에서 `WHERE FMI > 0`으로 제외하는 방식을 권장합니다.

## 3. 사전 준비

## 3.1 MySQL 설치 및 실행

로컬 PC 또는 서버에 MySQL이 설치되어 있어야 합니다.

MySQL 접속이 가능한지 먼저 확인합니다.

```bash
mysql -u root -p
```

정상적으로 접속되면 준비 완료입니다.

## 3.2 Python 패키지 설치

프로젝트 루트인 `fmi/` 폴더에서 아래 명령어를 실행합니다.

```bash
pip install pandas mysql-connector-python python-dotenv
```

가상환경을 사용하는 경우 예시는 다음과 같습니다.

```bash
python -m venv .venv
.venv\Scripts\activate
pip install pandas mysql-connector-python python-dotenv
```

## 4. `.env` 설정

프로젝트 루트의 `.env` 파일에 MySQL 접속 정보를 작성합니다.

예시:

```env
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=본인_mysql_비밀번호
MYSQL_DATABASE=kma_fire_risk
DROP_EXISTING_TABLES=false
BATCH_SIZE=50000
```

각 항목 의미는 다음과 같습니다.

| 항목 | 의미 |
|---|---|
| `MYSQL_HOST` | MySQL 서버 주소 |
| `MYSQL_PORT` | MySQL 포트 |
| `MYSQL_USER` | MySQL 사용자명 |
| `MYSQL_PASSWORD` | MySQL 비밀번호 |
| `MYSQL_DATABASE` | 생성할 데이터베이스 이름 |
| `DROP_EXISTING_TABLES` | 기존 테이블 삭제 후 재생성 여부 |
| `BATCH_SIZE` | 한 번에 적재하는 행 수 (기본값 50,000) |

`MYSQL_DATABASE`는 기상 DB와 동일하게 `kma_fire_risk`로 두면 같은 데이터베이스에 `forest_fmi` 테이블이 추가됩니다.

처음 실행하거나 기존 데이터를 유지하고 싶으면 다음처럼 둡니다.

```env
DROP_EXISTING_TABLES=false
```

테이블 구조를 수정했거나 기존 테이블을 완전히 새로 만들고 싶으면 다음처럼 바꿉니다.

```env
DROP_EXISTING_TABLES=true
```

주의: `DROP_EXISTING_TABLES=true`로 실행하면 기존 `forest_fmi` 테이블이 삭제되고 다시 생성됩니다. 이미 적재된 데이터도 함께 삭제됩니다.

## 5. 실행 순서

반드시 프로젝트 루트 폴더인 `fmi/`에서 실행하는 것을 권장합니다.

## 5.1 데이터베이스 및 테이블 생성

```bash
python db_query/create_database.py
```

이 명령어는 다음 작업을 수행합니다.

```text
1. .env에서 MySQL 접속 정보 읽기
2. MYSQL_DATABASE 이름으로 데이터베이스 생성 (없으면 신규 생성, 있으면 유지)
3. forest_fmi 테이블 생성
```

정상 실행되면 MySQL에 `forest_fmi` 테이블이 생성됩니다.

## 5.2 CSV 데이터 적재

```bash
python db_query/load_data.py
```

이 명령어는 다음 작업을 수행합니다.

```text
1. forest_fmi.csv 읽기 (562,276행)
2. 정수형 컬럼의 소수점 제거 (예: 2.0 → 2)
3. 결측값(NaN) → NULL 변환
4. MySQL forest_fmi 테이블에 데이터 적재 (50,000건씩 배치)
```

## 6. 데이터 적재 규칙

## 6.1 전체 데이터 적재

기상 DB와 달리 지역 필터링 없이 `forest_fmi.csv` 전체를 적재합니다.

`forest_fmi.csv`는 이미 강원도 전체 산림 구역(562,276개)만 포함하고 있기 때문입니다.

## 6.2 소수점 정수값 변환

원본 CSV에서 정수 컬럼이 `2.0`, `1.0`처럼 소수점 형태로 저장되어 있는 경우가 있습니다.

적재 코드에서 아래 컬럼을 자동으로 정수로 변환합니다.

```text
polygon_id, STORUNST, FROR_CD, FRTP_CD, FMI, AGCLS_CD, DMCLS_CD, HEIGHT, 갱신년도
```

예시:

```text
2.0  → 2
1.0  → 1
```

## 6.3 결측값 처리

아래 값은 모두 DB `NULL`로 처리됩니다.

```text
NaN, None, null, 빈 문자열
```

특히 `갱신년도` 컬럼은 일부 구역에서 결측이 존재하며, 해당 행은 `갱신년도 = NULL`로 적재됩니다.

## 6.4 중복 키 처리

동일한 `polygon_id`가 이미 존재하면 `ON DUPLICATE KEY UPDATE`로 기존 데이터를 갱신합니다.

스크립트를 여러 번 실행해도 중복 오류 없이 안전하게 동작합니다.

## 7. 테이블 설명

## 7.1 `forest_fmi` 임상도 및 임상지수

강원도 전체 산림 구역의 수종 구성, 임상 특성, 임상지수(FMI), 공간 좌표를 저장합니다.

전체 컬럼:

| 컬럼 | 타입 | 의미 |
|---|---|---|
| `polygon_id` | INT (PK) | 구역 고유 ID — 다른 테이블과 조인 시 key |
| `STORUNST` | TINYINT | 임목존재코드 (1=입목지, 2=무립목지, 0=비산림) |
| `FROR_CD` | TINYINT | 임종코드 (1=인공림, 2=천연림) |
| `FRTP_CD` | TINYINT | 임상코드 (1=침엽, 2=활엽, 3=혼효, 4=죽림, 0=무립목) |
| `FRTP_NM` | VARCHAR(20) | 임상명 |
| `FMI` | TINYINT | **임상지수** (침엽=10, 혼효=3, 활엽=2, 죽림=1, 비산림=0) |
| `KOFTR_NM` | VARCHAR(50) | 수종명 (소나무, 잣나무, 낙엽송 등) |
| `AGCLS_CD` | TINYINT | 영급코드 (1~9, 10년 단위) |
| `AGCLS_NM` | VARCHAR(20) | 영급명 (1영급=1~10년생) |
| `DMCLS_CD` | TINYINT | 경급코드 |
| `DMCLS_NM` | VARCHAR(20) | 경급명 (치수/소경목/중경목/대경목) |
| `DNST_CD` | CHAR(1) | 밀도코드 (A=소, B=중, C=밀) |
| `DNST_NM` | VARCHAR(10) | 밀도명 |
| `HEIGHT` | TINYINT | 임분고코드 (2m 단위) |
| `HEIGHT_NM` | VARCHAR(30) | 임분고명 |
| `갱신년도` | SMALLINT | 해당 구역 마지막 갱신 연도 (NULL 가능) |
| `Shape_Area` | DOUBLE | 구역 면적 (㎡) |
| `centroid_lon` | DECIMAL(10,6) | 중심점 경도 (WGS84) — 기상 관측소 매핑용 |
| `centroid_lat` | DECIMAL(10,6) | 중심점 위도 (WGS84) — 기상 관측소 매핑용 |

FMI 값 기준:

| 임상 유형 | FMI | 구역 수 | 비율 |
|---|---|---|---|
| 침엽수림 (소나무, 잣나무, 낙엽송 등) | 10 | 235,078 | 41.8% |
| 활엽수림 (참나무류, 신갈나무 등) | 2 | 216,491 | 38.5% |
| 혼효림 (침활 혼합) | 3 | 76,625 | 13.6% |
| 무립목지/비산림 | 0 | 33,070 | 5.9% |
| 죽림 | 1 | 1,012 | 0.2% |

## 8. 실행 후 확인 쿼리

MySQL에서 아래 쿼리로 데이터가 정상 적재되었는지 확인할 수 있습니다.

```sql
USE kma_fire_risk;

SHOW TABLES;

SELECT COUNT(*) FROM forest_fmi;
```

정상 적재 시 562,276이 출력됩니다.

FMI별 구역 수를 확인하려면 다음 쿼리를 실행합니다.

```sql
SELECT FMI, FRTP_NM, COUNT(*) AS cnt
FROM forest_fmi
GROUP BY FMI, FRTP_NM
ORDER BY FMI DESC;
```

산불 분석 대상(FMI > 0)만 확인하려면 다음 쿼리를 실행합니다.

```sql
SELECT COUNT(*) FROM forest_fmi WHERE FMI > 0;
```

특정 수종의 분포를 확인하는 예시는 다음과 같습니다.

```sql
SELECT KOFTR_NM, COUNT(*) AS cnt
FROM forest_fmi
WHERE FMI > 0
GROUP BY KOFTR_NM
ORDER BY cnt DESC
LIMIT 10;
```

기상 DB의 `station_info`와 공간 조인하는 예시는 다음과 같습니다.

```sql
SELECT f.polygon_id, f.FMI, f.centroid_lon, f.centroid_lat,
       s.STN_ID, s.STN_KO
FROM forest_fmi f
JOIN station_info s
  ON s.STN_ID = (
      SELECT STN_ID FROM station_info
      ORDER BY POW(s.LAT - f.centroid_lat, 2) + POW(s.LON - f.centroid_lon, 2)
      LIMIT 1
  )
WHERE f.FMI > 0
LIMIT 10;
```

주의: 위 공간 조인은 예시용입니다. 실제 분석에서는 Python의 `scipy.spatial.cKDTree`로 최근접 관측소를 매핑하는 방식을 권장합니다.

## 9. 자주 발생하는 문제

## 9.1 `Unknown database` 에러

데이터베이스가 아직 생성되지 않은 상태입니다.

먼저 아래 명령어를 실행합니다.

```bash
python db_query/create_database.py
```

## 9.2 `Access denied for user` 에러

`.env`의 MySQL 계정 정보가 잘못되었을 가능성이 큽니다.

다음을 확인합니다.

```text
MYSQL_USER
MYSQL_PASSWORD
MYSQL_HOST
MYSQL_PORT
```

## 9.3 CSV 파일을 찾을 수 없다는 에러

`load_data.py`는 스크립트 위치 기준으로 상위 폴더(`fmi/`)에서 `forest_fmi.csv`를 찾습니다.

아래 경로에 파일이 있는지 확인합니다.

```text
fmi/forest_fmi.csv
```

또한 명령어는 프로젝트 루트인 `fmi/`에서 실행하는 것을 권장합니다.

## 9.4 테이블 구조 변경 후 반영이 안 되는 경우

`CREATE TABLE IF NOT EXISTS`는 이미 존재하는 테이블의 컬럼 타입을 변경하지 않습니다.

`.env`에서 아래처럼 설정한 뒤 다시 실행합니다.

```env
DROP_EXISTING_TABLES=true
```

그리고 다시 실행합니다.

```bash
python db_query/create_database.py
python db_query/load_data.py
```

초기화 후 다시 보호 모드로 바꾸고 싶으면 `.env`를 수정합니다.

```env
DROP_EXISTING_TABLES=false
```

## 9.5 적재가 느린 경우

`BATCH_SIZE`를 늘리면 속도가 빨라질 수 있습니다. 메모리가 충분한 환경에서는 아래처럼 설정합니다.

```env
BATCH_SIZE=100000
```

반대로 메모리 부족 오류가 발생하면 줄입니다.

```env
BATCH_SIZE=10000
```

## 10. 권장 작업 흐름

처음 세팅하는 팀원은 아래 순서대로 실행하면 됩니다.

```bash
cd fmi
pip install pandas mysql-connector-python python-dotenv
python db_query/create_database.py
python db_query/load_data.py
```

기존 테이블을 초기화하고 다시 적재해야 할 경우:

```bash
# .env에서 DROP_EXISTING_TABLES=true로 수정 후
python db_query/create_database.py
python db_query/load_data.py
```

초기화 후 다시 기존 보호 모드로 바꾸고 싶으면 `.env`를 다시 수정합니다.

```env
DROP_EXISTING_TABLES=false
```
