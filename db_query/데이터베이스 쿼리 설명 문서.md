# WEATHER MySQL 데이터베이스 구축 및 데이터 적재 가이드

이 문서는 팀원이 동일한 방식으로 MySQL 데이터베이스를 생성하고, 기상 CSV 데이터를 적재할 수 있도록 작성한 실행 가이드입니다.

## 1. 프로젝트 폴더 구조

현재 프로젝트는 아래 구조를 기준으로 합니다.

```text
WEATHER/
├─ data/
│  └─ raw/
│     ├─ 강수량_20191101_20251231.csv
│     ├─ 기온_20191101_20251231.csv
│     ├─ 바람_20191101_20251231.csv
│     └─ 습도_20191101_20251231.csv
│
├─ db_query/
│  ├─ 데이터베이스 쿼리 설명 문서.md
│  ├─ create_database.py
│  └─ load_data.py
│
├─ temp/
│  ├─ AWS지점정보.csv
│  ├─ AWS지점정보.ipynb
│  ├─ getData.ipynb
│  └─ preprocess.ipynb
│
├─ .env
└─ .gitignore
```

주의: `기상통합_20191101_20251231.csv` 파일은 이 적재 과정에서 사용하지 않습니다.

## 2. 각 파일 역할

### `db_query/create_database.py`

MySQL 데이터베이스와 테이블을 생성하는 코드입니다.

생성되는 테이블은 총 5개입니다.

```text
station_info   관측소 기본정보
precipitation  강수량
temperature    기온
wind           바람
humidity       습도
```

기본키는 다음과 같습니다.

```text
station_info:
STN_ID, LAT, LON

precipitation, temperature, wind, humidity:
STN_ID, LAT, LON, TMA
```

### `db_query/load_data.py`

CSV 파일을 읽어서 MySQL 테이블에 데이터를 적재하는 코드입니다.

적재 대상 파일은 다음과 같습니다.

```text
temp/AWS지점정보.csv
data/raw/강수량_20191101_20251231.csv
data/raw/기온_20191101_20251231.csv
data/raw/바람_20191101_20251231.csv
data/raw/습도_20191101_20251231.csv
```

`temp/AWS지점정보.csv`에서 `LAW_ADDR` 또는 `RAW_ADDR` 값이 `강원특별자치도`로 시작하는 관측소의 `STN_ID`만 추출합니다.

이후 5개 테이블 모두 해당 `STN_ID`에 해당하는 데이터만 적재합니다.

## 3. 사전 준비

## 3.1 MySQL 설치 및 실행

로컬 PC 또는 서버에 MySQL이 설치되어 있어야 합니다.

MySQL 접속이 가능한지 먼저 확인합니다.

```bash
mysql -u root -p
```

정상적으로 접속되면 준비 완료입니다.

## 3.2 Python 패키지 설치

프로젝트 루트인 `WEATHER/` 폴더에서 아래 명령어를 실행합니다.

```bash
pip install pandas mysql-connector-python python-dotenv
```

가상환경을 사용하는 경우 예시는 다음과 같습니다.

```bash
python -m venv .venv
.venv\Scripts\activate
pip install pandas mysql-connector-python python-dotenv
```

## 4. `.env` 설정

프로젝트 루트의 `.env` 파일에 MySQL 접속 정보를 작성합니다.

예시:

```env
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=본인_mysql_비밀번호
MYSQL_DATABASE=weather

DROP_EXISTING_TABLES=false
```

각 항목 의미는 다음과 같습니다.

| 항목 | 의미 |
|---|---|
| `MYSQL_HOST` | MySQL 서버 주소 |
| `MYSQL_PORT` | MySQL 포트 |
| `MYSQL_USER` | MySQL 사용자명 |
| `MYSQL_PASSWORD` | MySQL 비밀번호 |
| `MYSQL_DATABASE` | 생성할 데이터베이스 이름 |
| `DROP_EXISTING_TABLES` | 기존 테이블 삭제 후 재생성 여부 |

처음 실행하거나 기존 데이터를 유지하고 싶으면 다음처럼 둡니다.

```env
DROP_EXISTING_TABLES=false
```

테이블 구조를 수정했거나 기존 테이블을 완전히 새로 만들고 싶으면 다음처럼 바꿉니다.

```env
DROP_EXISTING_TABLES=true
```

주의: `DROP_EXISTING_TABLES=true`로 실행하면 기존 5개 테이블이 삭제되고 다시 생성됩니다. 이미 적재된 데이터도 함께 삭제됩니다.

## 5. 실행 순서

반드시 프로젝트 루트 폴더인 `WEATHER/`에서 실행하는 것을 권장합니다.

## 5.1 데이터베이스 및 테이블 생성

```bash
python db_query/create_database.py
```

이 명령어는 다음 작업을 수행합니다.

```text
1. .env에서 MySQL 접속 정보 읽기
2. MYSQL_DATABASE 이름으로 데이터베이스 생성
3. station_info, precipitation, temperature, wind, humidity 테이블 생성
```

정상 실행되면 MySQL에 5개 테이블이 생성됩니다.

## 5.2 CSV 데이터 적재

```bash
python db_query/load_data.py
```

이 명령어는 다음 작업을 수행합니다.

```text
1. temp/AWS지점정보.csv 읽기
2. 강원특별자치도 관측소 STN_ID 추출
3. data/raw의 강수량, 기온, 바람, 습도 CSV 읽기
4. 강원특별자치도 STN_ID에 해당하는 데이터만 필터링
5. MySQL 테이블에 데이터 적재
```

## 6. 데이터 적재 규칙

## 6.1 강원특별자치도 데이터만 적재

`load_data.py`는 `temp/AWS지점정보.csv`에서 주소 컬럼을 확인합니다.

우선순위는 다음과 같습니다.

```text
1. LAW_ADDR 컬럼이 있으면 LAW_ADDR 사용
2. LAW_ADDR이 없고 RAW_ADDR 컬럼이 있으면 RAW_ADDR 사용
```

주소가 `강원특별자치도`로 시작하는 행만 필터링하고, 해당 행들의 `STN_ID`를 기준으로 데이터를 적재합니다.

## 6.2 REQUEST_ 컬럼 제외

API 수집 과정에서 생성된 다음과 같은 컬럼은 DB에 적재하지 않습니다.

```text
REQUEST_STN_ID
REQUEST_TM1
REQUEST_TM2
```

즉, `REQUEST_`로 시작하는 모든 컬럼은 자동으로 제외됩니다.

## 6.3 시간 컬럼 보정

기상청 데이터에서 발생시각 컬럼은 `0530`처럼 앞자리에 0이 붙는 값이 존재합니다.

CSV 저장/로드 과정에서 `0530`이 `530`으로 바뀌는 경우가 있기 때문에, 적재 코드에서 시간 컬럼을 4자리 문자열로 보정합니다.

예시:

```text
530  → 0530
30   → 0030
0    → 0000
0700= → 0700
```

또한 컬럼명이나 값 끝에 붙은 `=`도 제거합니다.

예시:

```text
RN_MAX_10M_OCUR_TMA= → RN_MAX_10M_OCUR_TMA
0700= → 0700
```

## 7. 테이블 설명

## 7.1 `station_info` 관측소 기본정보

관측소의 지점번호, 위도, 경도, 지점명, 법정동주소 등을 저장합니다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `STN_ID` | 지점번호 |
| `LAT` | 위도 |
| `LON` | 경도 |
| `STN_KO` | 지점명(한글) |
| `STN_EN` | 지점명(영문) |
| `LAW_ADDR` | 법정동주소 |

## 7.2 `precipitation` 강수량

일별 강수량 통계 데이터를 저장합니다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `TMA` | 날짜(yyyymmdd) |
| `RN_DSUM` | 합계강수량(mm) |
| `RN_MAX_1HR` | 1시간최다강수량(mm) |
| `RN_MAX_6HR` | 6시간최다강수량(mm) |
| `RN_MAX_10M` | 10분최다강수량(mm) |

## 7.3 `temperature` 기온

일별 기온 통계 데이터를 저장합니다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `TA_DAVG` | 평균기온(℃) |
| `TMX_DD` | 최고기온(℃) |
| `TMN_DD` | 최저기온(℃) |
| `MRNG_TMN` | 아침최저기온(℃) |
| `DYTM_TMX` | 낮최고기온(℃) |
| `NGHT_TMN` | 밤최저기온(℃) |

## 7.4 `wind` 바람

일별 바람 통계 데이터를 저장합니다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `WS_DAVG` | 평균풍속(m/s) |
| `WS_INS_MAX` | 최대순간풍속(m/s) |
| `WD_INS_MAX` | 최대순간풍속시풍향(degree) |
| `WS_MAX` | 최대풍속(m/s) |
| `WD_MAX` | 최대풍속시풍향(degree) |
| `WD_FRQ` | 최다풍향(degree) |
| `WS_MIX` | 합성풍속(m/s) |
| `WD_MIX` | 합성풍향(degree) |

## 7.5 `humidity` 습도

일별 습도 통계 데이터를 저장합니다.

주요 컬럼:

| 컬럼 | 의미 |
|---|---|
| `RHM_AVG` | 평균상대습도(%) |
| `RHM_MIN` | 최저상대습도(%) |
| `RHM_MIN_OCUR_TMA` | 최저상대습도 발생시각 |

## 8. 실행 후 확인 쿼리

MySQL에서 아래 쿼리로 데이터가 정상 적재되었는지 확인할 수 있습니다.

```sql
USE weather;

SHOW TABLES;

SELECT COUNT(*) FROM station_info;
SELECT COUNT(*) FROM precipitation;
SELECT COUNT(*) FROM temperature;
SELECT COUNT(*) FROM wind;
SELECT COUNT(*) FROM humidity;
```

강원특별자치도 관측소만 들어갔는지 확인하려면 다음 쿼리를 실행합니다.

```sql
SELECT STN_ID, STN_KO, LAW_ADDR
FROM station_info
WHERE LAW_ADDR LIKE '강원특별자치도%';
```

날짜별 데이터 확인 예시는 다음과 같습니다.

```sql
SELECT *
FROM wind
ORDER BY TMA, STN_ID
LIMIT 10;
```

## 9. 자주 발생하는 문제

## 9.1 `Data too long for column ...` 에러

예시:

```text
mysql.connector.errors.DataError: 1406 (22001): Data too long for column 'RN_MAX_10M_OCUR_TMA' at row 1
```

원인:

```text
- 발생시각 값에 0700=처럼 불필요한 = 문자가 붙어 있음
- 또는 4자리보다 긴 문자열이 들어옴
```

대응:

```text
- load_data.py에서 값 끝의 = 제거
- 시간 컬럼은 4자리 문자열로 보정
- 그래도 이상값이 있으면 해당 행을 확인해야 함
```

기존 테이블 구조가 이미 만들어진 상태라면 수정된 스키마가 반영되지 않을 수 있습니다. 이 경우 `.env`에서 아래처럼 설정한 뒤 다시 실행합니다.

```env
DROP_EXISTING_TABLES=true
```

그리고 다시 실행합니다.

```bash
python db_query/create_database.py
python db_query/load_data.py
```

## 9.2 `Unknown database` 에러

데이터베이스가 아직 생성되지 않은 상태입니다.

먼저 아래 명령어를 실행합니다.

```bash
python db_query/create_database.py
```

## 9.3 `Access denied for user` 에러

`.env`의 MySQL 계정 정보가 잘못되었을 가능성이 큽니다.

다음을 확인합니다.

```text
MYSQL_USER
MYSQL_PASSWORD
MYSQL_HOST
MYSQL_PORT
```

## 9.4 CSV 파일을 찾을 수 없다는 에러

현재 폴더 구조가 README와 같은지 확인합니다.

특히 아래 파일 위치를 확인합니다.

```text
temp/AWS지점정보.csv
data/raw/강수량_20191101_20251231.csv
data/raw/기온_20191101_20251231.csv
data/raw/바람_20191101_20251231.csv
data/raw/습도_20191101_20251231.csv
```

또한 명령어는 프로젝트 루트인 `WEATHER/`에서 실행하는 것을 권장합니다.
