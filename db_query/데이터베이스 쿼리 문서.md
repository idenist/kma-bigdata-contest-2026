# KMA 기상 데이터 MySQL 적재 가이드

이 문서는 팀원이 로컬 또는 서버 환경에서 MySQL 데이터베이스를 생성하고, 기상청 API Hub에서 수집한 CSV 데이터를 MySQL 테이블에 적재하기 위한 실행 방법을 정리한 문서입니다.

## 1. 프로젝트 목적

본 프로젝트는 강원특별자치도 관측소에 해당하는 기상 데이터를 MySQL 데이터베이스에 저장하기 위한 코드입니다.

구성되는 테이블은 총 5개입니다.

| 테이블명 | 설명 |
|---|---|
| `station_info` | 관측소 기본정보 |
| `precipitation` | 강수량 데이터 |
| `temperature` | 기온 데이터 |
| `wind` | 바람 데이터 |
| `humidity` | 습도 데이터 |

`AWS지점정보.csv`에서 `LAW_ADDR`이 `강원특별자치도`로 시작하는 관측소의 `STN_ID`를 추출한 뒤, 해당 관측소 데이터만 각 테이블에 적재합니다.

## 2. 파일 구성

프로젝트 폴더는 다음과 같이 구성됩니다.

```text
kma_mysql_project/
├── create_database.py
├── load_data.py
├── requirements.txt
├── .env.example
├── README.md
├── AWS지점정보.csv

├── 바람_20191101_20251231_sample.csv
└── 습도_20191101_20251231_sample.csv
```

각 파일의 역할은 다음과 같습니다.

| 파일명 | 역할 |
|---|---|
| `create_database.py` | MySQL 데이터베이스 및 5개 테이블 생성 |
| `load_data.py` | CSV 파일을 읽고 강원특별자치도 관측소 데이터만 DB에 적재 |
| `requirements.txt` | 필요한 Python 패키지 목록 |

## 3. 사전 준비

### 3.1 Python 설치

Python 3.10 이상 사용을 권장합니다.

버전 확인:

```bash
python --version
```

또는 환경에 따라:

```bash
python3 --version
```

### 3.2 MySQL 설치 및 실행

MySQL 서버가 실행 중이어야 합니다.

MySQL 접속 확인:

```bash
mysql -u root -p
```

접속이 되면 MySQL 서버 준비는 완료된 것입니다.

### 3.3 Python 패키지 설치

pip install -r requirements.txt

| 패키지 | 용도 |
|---|---|

## 4. 환경변수 설정

`.env.example` 파일을 복사해서 `.env` 파일을 만듭니다.

### Windows PowerShell

```powershell
copy .env.example .env
```

### macOS / Linux

```bash
cp .env.example .env
```

그 다음 `.env` 파일을 열어서 본인 MySQL 환경에 맞게 수정합니다.

예시:

```env
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=your_password
MYSQL_DATABASE=kma_weather
DROP_EXISTING_TABLES=false

STATION_FILE=AWS지점정보.csv
PRECIPITATION_FILE=강수량_20191101_20251231_sample.csv
TEMPERATURE_FILE=기온_20191101_20251231_sample.csv
WIND_FILE=바람_20191101_20251231_sample.csv
HUMIDITY_FILE=습도_20191101_20251231_sample.csv
```

각 항목의 의미는 다음과 같습니다.

| 항목 | 의미 |
|---|---|
| `MYSQL_HOST` | MySQL 서버 주소 |
| `MYSQL_PORT` | MySQL 포트 번호. 기본값은 3306 |
| `MYSQL_USER` | MySQL 사용자명 |
| `MYSQL_PASSWORD` | MySQL 비밀번호 |
| `MYSQL_DATABASE` | 생성할 데이터베이스 이름 |
| `DROP_EXISTING_TABLES` | 기존 테이블 삭제 후 재생성 여부 |
| `STATION_FILE` | 관측소 기본정보 CSV 파일명 |
| `PRECIPITATION_FILE` | 강수량 CSV 파일명 |
| `TEMPERATURE_FILE` | 기온 CSV 파일명 |
| `WIND_FILE` | 바람 CSV 파일명 |
| `HUMIDITY_FILE` | 습도 CSV 파일명 |

## 5. 실행 순서

반드시 아래 순서대로 실행합니다.

### 5.1 데이터베이스 및 테이블 생성

```bash
python create_database.py
```

이 코드는 다음 작업을 수행합니다.

1. `.env` 파일에서 MySQL 접속 정보를 읽습니다.
2. `MYSQL_DATABASE`에 지정된 데이터베이스를 생성합니다.
3. 다음 5개 테이블을 생성합니다.
   - `station_info`
   - `precipitation`
   - `temperature`
   - `wind`
   - `humidity`

기존 테이블이 이미 있고 새 테이블 정의를 반영해야 한다면 `.env`에서 다음처럼 설정합니다.

```env
DROP_EXISTING_TABLES=true
```

주의: `DROP_EXISTING_TABLES=true`로 실행하면 기존 5개 테이블이 삭제되고 다시 생성됩니다. 이미 적재된 데이터도 삭제됩니다.

---

### 5.2 CSV 데이터 적재

테이블 생성이 끝난 후 다음 명령어를 실행합니다.

```bash
python load_data.py
```

이 코드는 다음 작업을 수행합니다.

1. `AWS지점정보.csv`를 읽습니다.
2. `LAW_ADDR` 컬럼이 `강원특별자치도`로 시작하는 행만 필터링합니다.
3. 해당 행의 `STN_ID` 목록을 만듭니다.
4. 관측소 기본정보, 강수량, 기온, 바람, 습도 CSV에서 해당 `STN_ID`에 해당하는 데이터만 추출합니다.
5. 각 데이터를 MySQL 테이블에 적재합니다.

## 6. 기본키 구조

각 테이블의 기본키는 다음과 같습니다.

| 테이블명 | 기본키 |
|---|---|
| `station_info` | `STN_ID`, `LAT`, `LON` |
| `precipitation` | `STN_ID`, `LAT`, `LON`, `TMA` |
| `temperature` | `STN_ID`, `LAT`, `LON`, `TMA` |
| `wind` | `STN_ID`, `LAT`, `LON`, `TMA` |
| `humidity` | `STN_ID`, `LAT`, `LON`, `TMA` |

`STN_ID`만 기본키로 두지 않고 `LAT`, `LON`까지 포함한 이유는 관측소 위치 정보를 함께 식별 기준으로 사용하기 위함입니다.

강수량, 기온, 바람, 습도 테이블은 일별 데이터이므로 `TMA`도 기본키에 포함합니다.

## 7. 데이터 전처리 규칙

`load_data.py`에는 적재 전에 다음 전처리 로직이 포함되어 있습니다.

### 7.1 강원특별자치도 관측소만 필터링

`AWS지점정보.csv`에서 다음 조건을 만족하는 관측소만 사용합니다.

```text
LAW_ADDR이 '강원특별자치도'로 시작하는 행
```

### 7.2 `REQUEST_` 컬럼 제외

수집 과정에서 생성된 다음과 같은 컬럼은 DB 적재에 사용하지 않습니다.

```text
REQUEST_STN_ID
REQUEST_TM1
REQUEST_TM2
```

즉, `REQUEST_`로 시작하는 모든 컬럼은 자동으로 제외됩니다.

### 7.3 컬럼명 끝의 `=` 제거

기상청 API 응답 또는 CSV 저장 과정에서 다음처럼 컬럼명 끝에 `=`가 붙는 경우가 있습니다.

```text
RN_MAX_10M_OCUR_TMA=
```

이 경우 자동으로 다음처럼 정리합니다.

```text
RN_MAX_10M_OCUR_TMA
```

### 7.4 시간값 끝의 `=` 제거

값 자체에 `=`가 붙는 경우도 처리합니다.

예시:

```text
0700= → 0700
```

### 7.5 HHMM 시간 보정

발생시각 컬럼은 `HHMM` 형식이어야 합니다.

CSV 저장 또는 pandas 로딩 과정에서 다음처럼 앞자리 0이 사라질 수 있습니다.

```text
0530 → 530
0030 → 30
0000 → 0
```

적재 코드에서는 이를 다음처럼 보정합니다.

```text
530 → 0530
30 → 0030
0 → 0000
```

## 8. 자주 발생할 수 있는 오류

### 8.1 Data too long for column 오류

예시:

```text
mysql.connector.errors.DataError: 1406 (22001): Data too long for column 'RN_MAX_10M_OCUR_TMA' at row 1
```

원인은 보통 다음 중 하나입니다.

1. 시간값에 `=` 같은 불필요한 문자가 붙어 있음
2. `HHMM` 형식보다 긴 값이 들어옴
3. 기존 테이블이 예전 스키마로 생성되어 있음

해결 방법:

1. 최신 `load_data.py`를 사용합니다.
2. `.env`에서 `DROP_EXISTING_TABLES=true`로 설정합니다.
3. 다시 테이블을 생성합니다.

```bash
python create_database.py
python load_data.py
```

---

### 8.2 Unknown database 오류

예시:

```text
Unknown database 'kma_weather'
```

해결 방법:

`create_database.py`를 먼저 실행해야 합니다.

```bash
python create_database.py
```

그 후 다시 적재 코드를 실행합니다.

```bash
python load_data.py
```

---

### 8.3 Access denied 오류

예시:

```text
Access denied for user 'root'@'localhost'
```

해결 방법:

`.env` 파일의 MySQL 계정 정보를 확인합니다.

```env
MYSQL_USER=root
MYSQL_PASSWORD=your_password
```

비밀번호에 특수문자가 포함되어 있어도 따옴표 없이 그대로 작성하면 됩니다.

---

### 8.4 CSV 파일을 찾을 수 없는 경우

예시:

```text
FileNotFoundError: AWS지점정보.csv
```

해결 방법:

1. CSV 파일이 프로젝트 폴더 안에 있는지 확인합니다.
2. 파일명이 `.env`의 파일명과 정확히 일치하는지 확인합니다.
3. Windows에서는 확장자가 숨겨져 `AWS지점정보.csv.csv`가 되어 있지 않은지 확인합니다.

## 9. 적재 결과 확인

MySQL에 접속한 뒤 다음 쿼리로 데이터가 들어갔는지 확인할 수 있습니다.

```sql
USE kma_weather;

SELECT COUNT(*) FROM station_info;
SELECT COUNT(*) FROM precipitation;
SELECT COUNT(*) FROM temperature;
SELECT COUNT(*) FROM wind;
SELECT COUNT(*) FROM humidity;
```

강원특별자치도 관측소만 들어갔는지 확인하려면:

```sql
SELECT STN_ID, STN_KO, LAW_ADDR
FROM station_info
LIMIT 20;
```

특정 테이블 일부를 확인하려면:

```sql
SELECT *
FROM wind
LIMIT 10;
```

## 11. 주의사항

1. 기존 테이블을 유지하고 싶으면 `DROP_EXISTING_TABLES=false`로 둡니다.
2. 테이블 구조를 바꾼 뒤 다시 반영하려면 `DROP_EXISTING_TABLES=true`로 실행해야 합니다.
3. `DROP_EXISTING_TABLES=true`는 기존 데이터를 삭제하므로 신중하게 사용합니다.
4. 날짜 컬럼 `TMA`는 `YYYYMMDD` 문자열입니다.
5. 발생시각 컬럼은 `HHMM` 문자열입니다.
6. CSV를 직접 열어 저장하면 엑셀에 의해 `0530`이 `530`으로 바뀔 수 있으므로 주의합니다.
